수동 마이그레이션의 4대 실패

단일 page는 수동으로도 가능하다. 그러나 1,099 page × 4가지 실패 요인은 곱셈으로 작동해 6개월 안에 프로젝트를 정지시킨다.

01 · Scale

규모가 조직 수명을 초과
1,099 page × 2주 = 42 person-years. 순차 처리 시 5명 팀도 8년+. 인력 turnover로 완주 불가.

02 · Consistency

일관성 붕괴
수십 명이 동시 작업 → FSD·DDD 해석이 page마다 발산. 코드 리뷰 비용이 구현 비용을 초과.

03 · Knowledge loss

암묵 규칙 유실
PHP에 묻혀 있는 비즈니스 규칙 — 할인 정책, 권한, 예외 처리 — 가 rewrite 중 사라진다. 사고로만 발견.

04 · No verification

동일성 증명 없음
"동일 기능"을 보장할 체계적 방법 부재. QA의 sampling은 1,099 page를 cover 못 함.

AI 코딩 ≠ AI 엔지니어링

시장에는 두 가지 흔한 접근이 있다. 둘 다 마이그레이션 규모에서는 실패한다. 이유는 같다 — 검증과 인간 개입을 설계하지 않았다.

Harness Engineering

2025–26 업계가 합의에 도달한 개념. 모델 바깥의 전체 시스템 — context, tools, mechanical constraints, verification gates, feedback loop, human checkpoint — 을 한 단위로 묶어 설계하는 것.

왜 Harness인가

단일 prompt 튜닝 (Prompt Engineering) 만으로는 같은 page에 같은 결과가 나오지 않는다. 입력 컨텍스트를 더 잘 구성해도 (Context Engineering) hallucination이 줄 뿐 없어지지 않는다. 실행 → 측정 → 수정이라는 loop가 모델 외부에 있어야 신뢰 가능한 SW가 된다.

실험적 근거: MetaGPT — Executable Feedback 제거 시 코드 정확도 −4.2~5.4%p. AlphaCodium — test-loop 추가로 단일 LLM 대비 +22%p.

Prompt Engineering단일 호출 최적화 · 1~10% 영향
Context Engineering3-Layer Context · 무엇을 넣을 것인가
Harness EngineeringMCP · gates · 3-Tier feedback · Review Queue — 검증 가능한 SW로

Harness 4대 구성요소

  • Ground Truth Input707 aispec + MCP — LLM이 추측하지 않음
  • Mechanical ConstraintsFSD · DDD · OpenAPI · ESLint · Gradle
  • Verification GatesStep 2–8: compile · test · diff · graph
  • Feedback Loop3-Tier — Silent → Notify → Human
