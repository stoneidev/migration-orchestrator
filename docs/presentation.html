수동 마이그레이션의 4대 실패

단일 page는 수동으로도 가능하다. 그러나 1,099 page × 4가지 실패 요인은 곱셈으로 작동해 6개월 안에 프로젝트를 정지시킨다.

01 · Scale

규모가 조직 수명을 초과
1,099 page × 2주 = 42 person-years. 순차 처리 시 5명 팀도 8년+. 인력 turnover로 완주 불가.

02 · Consistency

일관성 붕괴
수십 명이 동시 작업 → FSD·DDD 해석이 page마다 발산. 코드 리뷰 비용이 구현 비용을 초과.

03 · Knowledge loss

암묵 규칙 유실

동일성 증명 없음

Harness Engineering

2025–26 업계가 합의에 도달한 개념. 모델 바깥의 전체 시스템 — context, tools, mechanical constraints, verification gates, feedback loop, human checkpoint — 을 한 단위로 묶어 설계하는 것.

왜 Harness인가

단일 prompt 튜닝 (Prompt Engineering) 만으로는 같은 page에 같은 결과가 나오지 않는다. 입력 컨텍스트를 더 잘 구성해도 (Context Engineering) hallucination이 줄 뿐 없어지지 않는다. 실행 → 측정 → 수정이라는 loop가 모델 외부에 있어야 신뢰 가능한 SW가 된다.

실험적 근거: MetaGPT — Executable Feedback 제거 시 코드 정확도 −4.2~5.4%p. AlphaCodium — test-loop 추가로 단일 LLM 대비 +22%p.

Prompt Engineering단일 호출 최적화 · 1~10% 영향
Context Engineering3-Layer Context · 무엇을 넣을 것인가
Harness EngineeringMCP · gates · 3-Tier feedback · Review Queue — 검증 가능한 SW로

Hub-and-Spoke Orchestration

Worker는 서로를 모른다. Orchestrator만이 artifact를 중재한다 — Andrew Ng (Multi-Agent 2024): "에이전트 간 자유 상호작용은 품질 예측이 불가능하다."

왜 Hub-and-Spoke인가

  • 관찰 가능성 — 모든 입출력이 한 곳을 지나므로 trace·cost·retry를 일관 관리
  • 교체 가능성 — Worker는 plug-and-play. Bedrock → Vertex 교체에 24h
  • 결정론적 부분 분리 — MCP는 LLM 아닌 정적 분석, 같은 인터페이스로 결합
  • 병렬 안전성 — 동일 page React/Java는 서로 모른 채 worktree에서 동시 실행

9-Step Pipeline (Page당)

한 page를 처리하는 사이클은 9 step의 고정된 시퀀스. 각 step은 schema-validated artifact를 출력하고, 다음 step의 입력으로 forward. 실패는 같은 step 안에서 3-Tier로 처리.

Phase A · Inception (1–2)aispec 로딩 + MCP gap 검증 · 결정론적 · 최대 5 page 병렬
Phase B · Contract (3)OpenAPI 생성 · Bedrock Sonnet · 5 page 병렬 (cache 적중률 높음)
Phase C · Construction (4–5)React/Java 코드 생성 · Claude CLI · worktree 격리 · 2 page 병렬 (CPU/git bottleneck)

Step 1–3은 사전 batch 가능, Step 4–8은 page별 직렬. 전체 처리 시간 = max(parallel) + serial이라 Phase C가 bottleneck.

9-Step Detail · Inception + Construction (1–5)

9-step pipeline 중 Step 1–5는 spec 로딩에서 React/Java 코드 생성까지. 각 step의 출력은 다음 step의 입력 schema이며, 모든 step에 verification gate가 inline으로 존재.

STEP 1
Spec Load

aispec.json 로딩 · JSON schema validation · review_score ≥ 0.85 게이트.

→ JSON schema · review threshold

STEP 2
MCP Gap Verify

php-analyzer로 spec 누락 점검: 함수 호출 그래프 · SQL · 미수집 권한 체크.

→ critical gaps = 0

STEP 3
OpenAPI Contract

Bedrock Sonnet이 aispec → openapi.yaml 생성. FE/BE 공통 계약. Few-shot은 Layer 1 cached.

→ openapi 3.x validator · breaking-change diff

STEP 4
React (FSD)

Claude CLI가 worktree/react에서 page slice 생성. Feature-Sliced-Design 컨벤션 강제.

→ ESLint · TypeScript strict · visual diff

STEP 5
Java (DDD)

Claude CLI가 worktree/java에서 Application Service · Aggregate · Repository 생성. Hexagonal port/adapter.

→ gradle compileJava · JUnit 컴파일

HUMAN CHECKPOINT
Review Queue (선택)

Step 4/5 출력이 visual diff 임계 초과 시 자동 진입. approve / refine prompt / skip page 옵션.

→ 평균 5–10% page만 진입

9-Step Detail · Verification + Operations (6–9)

Step 6–9가 SDLC 교과서에 없는 영역이다. "정말 동일한가"의 자동 증명과, 다음 page를 더 잘 처리하기 위한 학습 단계.

STEP 6
Java Test

aispec의 test_scenarios → JUnit 생성 · 실행. 실패 시 Tier 1 자동 재시도, 3회 시 Tier 2.

→ JUnit 100% pass · coverage ≥ 70%

STEP 7
Integration

FE/BE를 OpenAPI 계약으로 contract-test. Pact 또는 schemathesis. 누락 endpoint 자동 검출.

→ contract validator pass

STEP 8
Functional Equivalence

MCP가 PHP vs Java call graph · 발생 SQL · 응답 schema 비교. Greenfield에 없는 핵심 단계.

→ graph diff · SQL set 동일 · response schema match

STEP 9
Complete · Learn

git step commit 체인 · Pattern Library에 신규 패턴 등록 (예: sql_fetch → Repository) · cost·token log 기록.

→ artifact chain persisted · pattern added

FAILURE LOOP
3-Tier Self-Correction

Step 2–8 어느 게이트도 실패하면 Tier 1 → Tier 2 → Tier 3로 자동 escalate.

→ T1 ~80% · T2 ~15% · T3 5–10%

OUTPUT
Page bolt 완료

branch migration/<page> · 모든 artifact가 1:1 매핑되어 commit log로 추적 가능.

→ Dashboard에 ✓ 표시 · cost/duration 누적
